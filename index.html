对于这种设计的初衷，网上的解释是：http://www.zuidaima.com/share/1843292395834368.htm

MethodAccessor实现有两个版本，一个是Java实现的，另一个是native code实现的。Java实现的版本在初始化时需要较多时间，但长久来说性能较好；native版本正好相反，启动时相对较快，但运行时间长了之后速度就比不过Java版了。这是HotSpot的优化方式带来的性能特性，同时也是许多虚拟机的共同点：跨越native边界会对优化有阻碍作用，它就像个黑箱一样让虚拟机难以分析也将其内联，于是运行时间长了之后反而是托管版本的代码更快些。 为了权衡两个版本的性能，Sun的JDK使用了“inflation”的技巧：让Java方法在被反射调用时，开头若干次使用native版，等反射调用次数超过阈值时则生成一个专用的MethodAccessor实现类，生成其中的invoke()方法的字节码，以后对该Java方法的反射调用就会使用Java版。

异步消息

异步消息是系统间通信的主要解决方案，主要解决消息发送和处理之间速率不匹配，消息产生者和消费者之间解耦。
异步消息中有两个概念：broker 和 destination，即代理和目的地。为了实现生产和消费之间的解耦，生产者将信息交给broker接管，然后代理负责转发到指定的目的地。目的地一般有两种形式：一种是队列，还有一种是主题。queue主要用于点对点消息通信，topic用于发布/订阅式的消息通信。
queue确保每条消息有唯一的发送者和接收者，topic可以有多个消息接收者共同监听主题。
企业级的消息代理实现由JMS和AMQP两种规范，后者兼容JMS并且夸语言平台，所以AMQP得使用比较广泛，RabbitMQ是AMQP规范的一种消息代理实现。

RabbitMQ


RabbitMQ里也有上面说到的基本的异步消息概念，多做了一层抽象, 在Producer和queue之间, 加入了交换器 (Exchange)。使Producer和queue解耦，交换器根据调度策略把消息给队列。
从而衍生除了几个重要的概念:

  • VirtualHost
  • Exchange
  • binding

virtual host是用户在RabbitMQ中权限控制的最小粒度，多个虚拟主机可以方便管理消息之间的权限问题，一个virtual host有一组exchange, queue, binding. 这样不同的virtual host之间是不能访问对方的消息的。

##Guava 不可变集合
最近使用了Guava这个库，其中的collection对于日常的Java开发简化了很多，当然也包括了其他很多包括缓存，IO，原型支持，并发等一系列的特性。
主要的一些特性包括：
避免使用了null, 简化实现Object方法，比如hashCode()和toString()，更好的异常处理。

Guava的Collections包括一些不可变集合，还包括一些新的集合类型如：multisets, multimaps, tables, bidirectional maps…
而且对Collections的扩展更方便，比如实现Iterator的方式更简单了。

ImmutableSet可以保证安全调用，并且避免了多线程是资源竞争的问题，使用时可以作为一个常量来处理。

Collections.unmodifiable 虽然也有类似的实现，但是只有当没有对这个集合类的引用的时候它才是真的安全的。

所以当我们不想改变一个集合的时候，可以用ImmutableXXX.copyOf 保护性的拷贝给一个Immutable 集合类。

在使用ImmutableSet.of()初始化一个不可变集合的时候，在构造时就是有序的，ImmutableSet.of("a", "b", "c", "a", "d", "b")顺序读取后会返回 “a”, “b”, “c”, “d”。

Immutable. copyOf()会提供自动的类型转换，

如果调用 ImmutableList.copyOf(foobar)会自动的返回foobar.asList()，把ImmutableSet转换为ImmutableList，不过由于数据结构的关系，这种转化无法在O(1)时间内完成。

Exception translation

发表于 2017-03-20

当在DAO中使用Hibernate，JPA，JDO时，要处理一些持久层技术的原生异常类型，比如HibernateException,PersistenceException,JDOException，取决于使用哪种持久层技术。这些运行时的异常不用声明或者catch。Spring提供了一种透明的异常类型转译的方法，通过@Repository注解。

预处理器ersistenceExceptionTranslationPostProcessor会自动的扫描到所有的异常转译类，这些类都实现了PersistenceExceptionTranslator接口，并且会根据@Repository注解的DAO类的抛出异常选择恰当的异常处理方式。

当然我们可以不用选择定制直接使用原本的exception translator，但这对理解Spring的ORM技术是有帮助的。

Redis 初学

发表于 2017-03-20

###什么是Redis？
Redis是一个key-value的数据库，和memcached类似，但是除了可以存储String类型之外，还可以存储lists，sets,和ordered sets,这些数据类型可以通过原子的push/pop来操作，还包括add/remove, union，intersection，求集合不同，等等操作都是原子操作。

###Jedis
Jedis数据库连接池：
JedisPool pool = new JedisPool(new JedisPoolConfig(), "localhost");
也可以声明成静态对象，这是线程安全的。
JedisPool 基于Commons Pool 2 所以可以通过和Commons Pool类似的方式配置。
在实际应用中：

Redis 提供主从复制的机制，一个已经声明的Jedis连接实例可以通过两种方式声明主从关系
一种是通过xml配置文件的方式，另外一种是直接通过jedis.slaveOf("host_name","port")声明。
从Redis2.6开始从数据库默认是只读的，写需求会报错。就算修改从的设置，写请求不会备份，而且会被重写覆盖。
当主机down掉后，可以把从机改成master，可以通过slave.slaveOfNoOne()修改slave成为master，如果有不止一台slave，其他slave要通过slaveOf修改主从关系。

###SDR(Spring Data Redis)
SDR提供了很高层的抽象，所以可以很方便的使用redis。
使用Redis的第一步是通过IoC容器连接redis数据库。一般是org.springframework.data.redis.connection这个包。RedisConnection提供了和Redis交互的句柄，也可以把Redis的异常信息转换成Spring的consistent DAO exception hierarchy。
RedisConnection通过RedisConnectionFactory来生成，这个工厂方法有类似于PersistenceExceptionTranslator的作用
